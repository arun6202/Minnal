## 0. Position

Minnal is an execution-first AI-native API workspace. It is not a Postman clone and not an observability dashboard.

The approved direction is a **workbench**:

- Request execution is always close at hand.
- AI is contextual, evidence-backed, and permissioned.
- Semantic structure replaces folder-first navigation over time.
- Offline/local trust is visible at all times.
- The graph helps understanding; it does not hide the request builder.

The Stitch artifact is visual inspiration for the Semantic Canvas and Why drawer only. It is not the whole shell.

---

## 1. Product Principles

1. **Execution before spectacle**
   A developer must be able to import, send, inspect, explain, search, and promote without hunting through a dashboard.

2. **Why everywhere**
   Any response, hook, auth decision, search result, or AI draft can expose its evidence and reasoning in the same right-side drawer.

3. **AI drafts; users approve**
   AI can explain, draft, search, classify, and dry-run. It cannot silently write persisted state, change auth, run hooks, or touch network without explicit user action.

4. **Local trust is visible**
   The model, quantization, offline state, memory status, environment, and hook safety state are persistent UI facts, not hidden settings.

5. **Typed-DAG mental model**
   Collections are not just folders. Folder parity is allowed for import familiarity, but Minnal's native view is clusters, dependencies, auth flows, and intent.

## 2. Shell Layout

```
┌─────────────────────────────────────────────────────────────────────────────┐
│ Top Intent Bar: command/search/draft + active env + quick import/send state │
├───────────────┬───────────────────────────────────────────────┬─────────────┤
│ Left Rail     │ Center Workbench                              │ Why Drawer  │
│               │                                               │             │
│ Workspace     │ Request Surface / Response Thread             │ Explanation │
│ Imports       │ Semantic Canvas mode                          │ Evidence    │
│ Collections   │ Playground mode                               │ Diff        │
│ Clusters      │ Search Results mode                           │ Hook review │
│ History       │                                               │ Actions     │
├───────────────┴───────────────────────────────────────────────┴─────────────┤
│ Status Strip: gemma-4-e4b | Q4/Q8 | offline | memory | env | hook safety    │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 2.1 Top Intent Bar

The top bar is the primary AI entry point and global search surface.

Examples:

- `Create a login request against staging`
- `Find token refresh failures`
- `Explain the last 401`
- `Generate an auth pre-hook for expired bearer tokens`
- `Search requests that create invoices`

Rules:

- It returns drafts or navigable results, not surprise side effects.
- Drafts land in Playground unless the user explicitly promotes.
- Search results include evidence: matched request, endpoint, response/status, embedding/keyword rationale.

### 2.2 Left Rail

The left rail is compact and stable.

Primary items:

- Workspace
- Import
- Requests
- Semantic Clusters
- Playground
- History
- Hooks
- Exports

Avoid nested folder-tree dominance. Imported Postman folders can appear inside Requests, but Semantic Clusters should become equally prominent once embeddings exist.

### 2.3 Center Workbench

The center area has modes, not unrelated pages.

Core modes:

- Request Workbench
- Semantic Canvas
- Search Results
- Playground
- Hook Review

The Request Workbench is the default when a request exists. The Semantic Canvas is a first-class mode, not a permanent replacement for execution.

### 2.4 Why Drawer

The right drawer is contextual and reusable.

Drawer states:

- Response explanation
- Auth diagnosis
- Semantic search rationale
- Hook dry-run and capability review
- Schema/OpenAPI comprehension
- Memory/model status explanation

Every drawer state must show:

- Claim
- Evidence
- Confidence or limitation
- Available action
- Whether action persists data or touches network

### 2.5 Status Strip

Persistent bottom strip:

```
gemma-4-e4b | Q4_K_M/Q8_0 | offline | memory ok | staging | hooks: gated
```

Rules:

- Never hide offline/model state.
- Memory pressure changes are visible and non-destructive.
- No silent model downgrade.
- Hook state must be legible before sending requests with pre-hooks.

## 3. V1 Screens

### 3.1 Empty Workspace

Goal: get the user to useful data fast.

Visible actions:

- Import Postman collection
- Import OpenAPI/Swagger spec
- Create request
- Ask Minnal to draft request

Empty state text should be short and operational. No marketing hero.

### 3.2 Request Workbench

Required above the fold:

- Method selector
- URL input
- Auth state
- Environment selector
- Send button
- Headers/body tabs
- Latest response status, latency, size
- Response body
- Why button/drawer affordance
- History thread for the current request

Behavior:

- Send is explicit.
- Auth hooks run only if approved.
- Failed responses offer Why, retry, save example, and compare with previous responses.

### 3.3 Semantic Canvas

Purpose: show relationships and intent clusters.

Node types:

- Request
- Collection
- Auth flow
- Environment
- Response anomaly
- Hook
- OpenAPI operation

Interactions:

- Select node opens a compact inspector.
- Double-click request opens Request Workbench.
- Why opens drawer with cluster rationale.
- Graph can be filtered by method, status family, auth scheme, environment, or recency.

Constraint:

- The graph is navigational and explanatory. It is not the only way to execute requests.

### 3.4 Semantic Search Results

Search by intent, not folder.

Each result shows:

- Request name or generated summary
- Method + URL
- Why it matched
- Last status/latency
- Collection/cluster
- Open in Workbench

Results should make embedding-based matches inspectable enough to trust.

### 3.5 Playground

Purpose: ephemeral AI-assisted drafting.

Rules:

- Zero SQLite writes until Promote.
- Clear `Not saved` state.
- Promote action is visually distinct and explicit.
- AI-created requests, headers, bodies, and hooks land here first.

### 3.6 Hook Review

Required sections:

- Natural-language request that produced the hook
- Generated source/WASM summary
- Capability grants
- Allowed hosts
- Dry-run output
- Diff/modifications
- Approve / Reject

Approving a hook is a security decision. The UI should feel like code review plus capability consent, not a casual confirm dialog.

### 3.7 Memory Pressure

States:

- Normal: `memory ok`
- Pressure detected: hold model until restart, show non-blocking drawer/message
- Below floor: block with clear explanation

Never silently unload, downgrade, or swap model mid-session.

## 4. Visual Direction

Keep from Stitch:

- Dark dense cockpit feel
- Tonal layers instead of heavy shadows
- Graph/canvas vocabulary
- Right-side evidence drawer
- Compact code/diff blocks

Adjust for Minnal:

- Less dashboard language
- Less glow/pulse
- More request/response affordance
- AI as integrated helper, not separate destination
- Stronger local/offline trust signal

Color semantics:

- Blue: selected state, navigation, neutral primary action
- Teal: AI-generated suggestion or insight
- Orange: Promote, approve, persisted/irreversible action
- Red: error, failed auth, unsafe capability, destructive risk
- Gray: inactive, metadata, low-emphasis chrome

Shape:

- 4px to 8px radius max for controls and panels
- Avoid pill-heavy UI except compact status chips
- No decorative gradient orbs
- No marketing hero layout

Typography:

- Monospace for URLs, headers, JSON, diffs, logs
- Sans-serif for chrome and AI explanations
- Compact labels, no oversized dashboard headings inside work surfaces

---

## 5. Interaction Rules

1. **Any AI result must expose evidence.**
2. **Any persistence must be explicit.**
3. **Any network-touching action must be user-triggered.**
4. **Any generated hook requires dry-run before approval.**
5. **Any response explanation must use stored context when available.**
6. **Any semantic search result must be openable as a normal request.**
7. **Any model/memory state change must be visible.**

## 6. Open Questions for Arun

1. Should the default non-empty workspace open to the last request or the last semantic cluster?
   Recommendation: last request, with cluster context visible.

2. Should the left rail use labels always, or collapse to icons at narrow widths?
   Recommendation: labels on desktop, icon rail on narrow layouts.

3. Should orange be reserved only for Promote/Approve, or also for the Never Settle identity mark?
   Recommendation: reserve orange for Promote/Approve so it keeps behavioral meaning.

4. Should schema drift appear in v1 UI at all?
   Recommendation: only as an OpenAPI comprehension/search explanation, not a primary v1 feature.

5. Should the graph be implemented in WinUI-native drawing or WebView2 if Path B wins?
   Recommendation: decide after Spike A; keep the data contract independent of rendering path.

## 7. Codex Scaffold Target

When Phase 4 begins, Codex should scaffold only the shell and static states first:

1. App shell with top intent bar, left rail, center workbench, right drawer, status strip.
2. Static Request Workbench with fake request/response.
3. Static Why drawer with response explanation and evidence block.
4. Static Playground with Promote disabled/enabled states.
5. Static Hook Review with capabilities and dry-run output.
6. Static Semantic Canvas placeholder with typed node contracts.

No real API execution, AI inference, hook runtime, or SQLite writes in the first UI scaffold PR.

---

## 8. Review Gate

Fail the UI scaffold if any of these happen:

- The first screen is a marketing/landing page.
- Request execution is hidden behind a graph-only dashboard.
- AI appears only as a separate tab.
- Offline/model state is missing from persistent chrome.
- Playground can write without explicit Promote.
- Hook approval lacks capability review.
- The app looks like Postman with a chat box attached.
